# OPNsense Configuration Guide

How the network side of a Zonclave deployment is built on OPNsense: one VLAN and one WireGuard tunnel per PPSK group, with a firewall policy that fails closed rather than leaking traffic.

This document covers the general pattern - the reusable shape of the configuration, independent of any one site. For the exact, site-specific steps (real interface names, subnets, and hardware) for this project's actual deployment, see [docs/runbook/phase1-opnsense-unifi.md](runbook/phase1-opnsense-unifi.md). For the full narrative install guide, see [installation-guide.md](installation-guide.md). For the authoritative spec, see [CLAUDE.md](../CLAUDE.md) Sections 5, 6, 9, 10, 11, and 12.

## 1. Scope

The Zonclave installer provisions one host: PostgreSQL, FreeRADIUS, and the admin panel. It does not touch OPNsense or the UniFi controller - those are separate appliances, and this side is configured manually, once per router, following the pattern below (CLAUDE.md Section 24.2).

That separation is deliberate: FreeRADIUS only ever does authentication and VLAN assignment, OPNsense only ever does routing, firewalling, and VPN policy. Credential logic never lives on the router, and routing logic never lives in RADIUS (CLAUDE.md Section 1).

Exact interface names, IP ranges, and physical port assignments are specific to each site's hardware and existing network - confirm them on the box itself before making changes, never assume they match a previous deployment.

## 2. Naming convention

Every PPSK group's network artifacts share one predictable naming pattern, so a search for a VLAN number surfaces everything related to it (CLAUDE.md Section 6):

| Item | Convention | Example |
| --- | --- | --- |
| VLAN ID | Reserved block, per deployment | `VLAN300` |
| WireGuard interface | `WG_VLAN<id>` | `WG_VLAN300` |
| Gateway | `GW_WG_VLAN<id>` | `GW_WG_VLAN300` |
| Firewall alias (subnet) | `NET_VLAN<id>` | `NET_VLAN300` |
| Firewall rule (allow) | `ALLOW_VLAN<id>_TO_GW_WG_VLAN<id>` | `ALLOW_VLAN300_TO_GW_WG_VLAN300` |
| Firewall rule (block) | `BLOCK_VLAN<id>_TO_RFC1918` | `BLOCK_VLAN300_TO_RFC1918` |

## 3. Per-group setup

Repeated identically for every PPSK group being provisioned on a router:

1. **VLAN interface** - a tagged sub-interface on the LAN-facing trunk, one per group, with its own `/24` and DHCP range.
2. **WireGuard tunnel** - a dedicated tunnel to that group's residential VPN peer, never shared with another group.
3. **Gateway** - created against that WireGuard interface, with monitoring enabled where the peer supports it, so OPNsense actually knows when the tunnel is down.
4. **Firewall rules, in this exact order** (first match wins):
   1. Block that VLAN's subnet to all private address space (RFC1918), with narrow, IP-scoped exceptions for DNS, RADIUS, and DHCP only.
   2. Allow that VLAN's subnet out, with the gateway explicitly pinned to that group's WireGuard tunnel - never a default route.

   The block rule must sit above the allow rule. This ordering is the entire mechanism that keeps one group's devices from ever reaching another group, the management network, or the plain internet connection (CLAUDE.md Section 10).

## 4. DNS through the tunnel

Each group's DNS queries are forced through that same group's WireGuard tunnel, not resolved centrally. A DNS leak is the most common way a "secure" tunnel setup quietly stops being secure - the traffic itself might be tunneled correctly while DNS queries go out the plain connection and reveal exactly what's being tunneled. Verify this with an actual leak test per group, not just by reasoning about the firewall rules (CLAUDE.md Section 11).

## 5. Fail-closed, not fail-open

If a group's WireGuard tunnel drops, that group's traffic is dropped - not silently rerouted onto the plain internet connection. This depends on there being no fallback rule anywhere in the ruleset that would let a VLAN reach the internet through anything other than its pinned gateway. OPNsense ships a default "allow LAN to any" rule on a fresh install; that default is explicitly removed or never applied to these VLAN interfaces (CLAUDE.md Section 12).

Verify by deliberately killing a tunnel and confirming a connected device loses internet access entirely, rather than falling back to the router's real public IP.

## 6. The UniFi side

- **RADIUS profile** - points the access points at the FreeRADIUS server and shared secret generated by the installer.
- **SSID** - configured for RADIUS-assigned PPSK, with the VLAN left unpinned at the SSID level; FreeRADIUS assigns it per credential via a RADIUS attribute, not a fixed network on the SSID. As of current UniFi firmware, this feature works on WPA2 (2.4GHz and 5GHz) - WPA3 and 6GHz aren't yet supported for RADIUS-assigned PPSK, worth confirming against the specific site's UniFi Network application version before committing to it (CLAUDE.md Section 8.3).
- **Trunk port** - the switch port feeding the router carries every provisioned VLAN tagged.

## 7. Verifying the result

A deployment is only proven once a real device connects with a PPSK, lands on the correct VLAN, and its outbound public IP matches that VLAN's residential tunnel - checked from the device itself, not inferred from the configuration. See [commands-reference.md](commands-reference.md) for the exact verification commands (`wg show`, `curl -s ifconfig.me`, `radtest`) used at each step, and CLAUDE.md Section 21.1 for the full acceptance test list.
